# Forensic Production-Readiness Audit: ChravelApp Load Failure

## 1. Executive Summary
The critical root cause preventing the application from loading across multiple environments is **an infinite page reload loop** triggered by an over-aggressive auto-recovery mechanism for chunk loading errors.

In `src/App.tsx` and `src/components/ErrorBoundary.tsx`, any unhandled promise rejection containing the string `"Failed to fetch"` (or `"Loading chunk"`) forces the application to automatically call `clearAllCaches()` and then `safeReload(true)`. Because the state flag preventing immediate reloads (`toastShown`) is reset upon every page reload, if an asset legitimately fails to load (due to a 404, bad base path deployment, or temporary network issue), the application will instantly reload the page endlessly. This creates a "blank screen" or rapid flickering effect, completely blocking the app from booting across all environments (Vercel, Render, PWA, IONOS, Lovable).

Additionally, there have been a massive number of Supabase database migrations (over 60 SQL files added in recent weeks) and updates to Edge Functions (e.g., `gmail-import-worker`, `smart-import`, etc.) that were merged into the codebase but likely lack the corresponding operational steps required to deploy them to production, causing subsequent backend/API failures once the app actually loads.

## 3. PR Follow-Through Audit
*Merged PRs over the last 7-10 days include significant schema and backend changes:*
- **Gmail Smart Import & OCR**: Added `20260401000000_smart_import.sql`, `20260402000000_gmail_token_expires.sql`. Requires running Supabase migrations and deploying Edge Functions (`gmail-auth`, `gmail-import-worker`).
- **Push Notification Fixes**: Added `20260407000000_fix_push_notification_rls.sql`. Requires running Supabase migrations.
- **Polls & Tasks Security**: Modifies RLS and adds new RPC functions.
**Risk:** If these migrations were not executed manually in the production dashboard (or via CI/CD), the application will experience persistent API 500/403 errors, degraded functionality, and potentially unhandled rejections that (before the fix in `App.tsx`) could trigger the infinite reload loop.

## 4. Required Non-Code Actions
1. **Run Supabase Migrations**: Execute all pending SQL files from `supabase/migrations/` against the production Supabase database.
2. **Redeploy Edge Functions**: Run `supabase functions deploy` for all edge functions to ensure the serverless environment matches the frontend expectations.
3. **Secret Verification**: Ensure `GMAIL_TOKEN_ENCRYPTION_KEY`, `GOOGLE_CLIENT_ID`, and `GOOGLE_CLIENT_SECRET` are properly set in Supabase Edge Function Secrets (not `.env`), as they are required by the new Smart Import features.
4. **Vercel / Render SPA Configuration**: Verify that all hosting providers are configured to rewrite all paths to `index.html` to prevent 404s on deep links, which would otherwise trigger the chunk loading error.

## 5. Minimal Recommended Fix Plan
1. **First Validate**: Run `npm run validate` locally. (Temporarily install `@eslint/js` if necessary to pass checks).
2. **Then Fix**: Apply a session storage cap (e.g., maximum 2 reloads per session) to the `safeReload()` loops in `App.tsx` and `ErrorBoundary.tsx`. If a chunk error occurs a third time, render the fallback error UI instead of forcing a reload. *(Note: This step has already been completed in the current codebase).*
3. **Then Redeploy**: Commit the updated `App.tsx` and `ErrorBoundary.tsx` and deploy to Vercel/Render/IONOS.
4. **Then Verify**: Execute the database migrations and verify the application successfully boots and does not loop on load.

## 6. Verification Plan
- **Local**: Run `npm run build && npm run preview`. Disconnect internet or block a JS chunk in Chrome DevTools Network tab. Verify the app attempts to reload exactly 2 times and then displays an error message, rather than looping infinitely.
- **Vercel / Render**: Deploy the fix. Navigate to a deep link (e.g., `/join/123`). Ensure the app loads without flashing or redirecting infinitely.
- **IONOS**: Verify `.htaccess` or server config correctly routes to `index.html`.
- **PWA**: Clear local caches manually via DevTools. Reload the app. Ensure the Service Worker registers and caches correctly without falling into an update loop.
- **Lovable Preview**: Open the app in Lovable. Verify that `isLovablePreview()` correctly bypasses the Service Worker and that chunk errors do not force a constant page refresh.
