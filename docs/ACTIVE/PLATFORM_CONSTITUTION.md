# Chravel Platform Constitution: Scalability, Concurrency & Integrity

**Date:** May 2026
**Status:** Canonical Platform Governing Architecture
**Audience:** Platform Infrastructure, Product Engineers, AI Assistants

---

## 1. Executive Summary

**The current state of Chravel's architecture is a fragmented mix of robust foundational infrastructure and dangerous, early-stage rapid-prototyping debt.**
The core stack (React, Supabase, TanStack Query, Capacitor) is highly capable. However, as Chravel evolves from simple friend trips (2-20 users) to pro trips and events (1K+ users), the platform’s handling of concurrency, state ownership, and distributed mutation is breaking down.

- **Safest Surfaces:** Auth, Edge Function JWT validation, core Supabase RLS, and Stream chat integration.
- **Most Dangerous Surfaces:** Unversioned shared state (calendar, tasks, basecamp), AI concurrent mutations, legacy Supabase chat triggers, and free-tier rate-limiting.
- **Coherence vs Fragmentation:** RLS policies are strong but naming conventions are fragmented. Shared mutations are dangerously fragmented (some use optimistic last-write-wins, some use versioned RPCs).

---

## 2. Full Platform System Map

- **Data Boundaries:**
  - **Supabase PostgreSQL:** Source of truth for relational state, membership, roles, core objects.
  - **Stream Chat:** Source of truth for realtime messaging and presence.
- **Ownership Model:**
  - `User` > `Trip` (Friend / Pro / Event) > `Channel` / `Object` (Task/Poll/Calendar).
- **Trust Boundaries:**
  - Client devices are **untrusted**.
  - Edge Functions are **trusted** (must validate user via `getBearerToken`).
  - Webhooks are **untrusted** (require signature verification).
- **Shared State Boundaries:**
  - Trip-level singletons (Basecamp, Cover).
  - Trip-level collections (Tasks, Polls, Events, Artifacts).
- **Realtime Boundaries:**
  - Supabase Realtime (moderate volume: objects, typing, status).
  - Stream Realtime (high volume: messages, reactions).
- **AI Mutation Boundaries:**
  - Gemini/Edge Functions acting on behalf of users must pass idempotency keys and version checks to prevent race conditions.
- **Scale Bottlenecks:**
  - DB trigger cascades (e.g., `sync_channel_memberships`).
  - N+1 fetching in legacy hooks.
  - O(N) permission calculations on large event trips.

## 3. Platform Invariants

1. **Strict Snake_Case DB / CamelCase Client:** Database columns *must* be snake_case. Client types *must* be camelCase. Mapping occurs at the adapter edge.
2. **Never Trust Client Timestamps:** All canonical timestamps (`created_at`, `updated_at`) are generated by Postgres default/triggers.
3. **Optimistic Concurrency Control (OCC) is Mandatory:** All shared mutable objects (Tasks, Calendar, Polls) *must* include an integer `version` field. Updates must use RPCs (`update_with_version`) to reject stale writes.
4. **Idempotency Keys for all Non-Idempotent Actions:** AI and Client creates/mutations must supply an `idempotency_key` (UUID) to prevent double-charging or duplicate entity creation.
5. **No Silent Last-Write-Wins:** Concurrent overwrites must fail loudly on the client, prompting a refresh/merge.
6. **Soft Deletes for Shared State:** Critical shared objects (Trips, Payments, Channels) use soft-deletion (`deleted_at`) to preserve audit history and prevent sync cascading failures.
7. **Canonical Membership Source:** `useTripMembersQuery` via Supabase is the sole truth for membership. Do not deduce state from cached client arrays.

---

## 4. Object Scope Constitution

- **User-Private:** User settings, payment methods, personal default notifications. (Direct user_id RLS).
- **Trip-Shared:** Calendar events, Tasks, Polls. (Trip member RLS; bounded to max 500 members).
- **Channel-Scoped:** Stream messages, reactions. (Channel participant RLS/Stream roles).
- **Event-Wide:** Announcements, Pro tier schedules. (Read-heavy, write-restricted to Event Admins).
- **Admin-Only:** Audit logs (`security_logs`), enterprise billing configs, API keys.
- **Ephemeral:** Typing indicators, active user presence (Stream/Supabase Presence).
- **Durable:** Payments, Artifacts, Canonical chat history.

---

## 5. Permission Model Constitution

- **Base RLS:** Every table must have an RLS policy. No `public` schemas exposed without RLS.
- **Role Hierarchy:** `Creator` > `Trip Admin` > `Member` > `Viewer/Guest`.
- **RBAC Single Source of Truth:** `user_trip_roles` and `channel_role_access`. Legacy boolean columns (`is_admin`) are deprecated.
- **AI/Concierge Scoping:** AI agents inherit the strict permissions of the invoking user. No "super-AI" bypasses unless explicitly configured as a system task.
- **Edge Function Authorization:** Require Authorization headers and `supabase.auth.getUser()`. Never trust a `user_id` passed in the JSON payload.

---

## 6. Concurrency + Mutation Constitution

- **Read Path:** TanStack Query `useQuery` with strict query keys (`[domain, tripId, entityId]`).
- **Write Path:** TanStack Query `useMutation` calling typed Supabase RPCs.
- **Conflict Resolution:**
  1. Client sends `idempotency_key` and `version: current_version`.
  2. RPC checks `version = current_version`.
  3. On mismatch, throw `409 Conflict`.
  4. Client intercepts `409`, triggers `queryClient.invalidateQueries()`, and alerts user "Data changed by someone else."
- **AI Batching:** AI generated objects (e.g., smart import) must use bulk RPC inserts inside a single Postgres transaction.

---

## 7. Access Funnel Constitution

- **Invite Links:** Must be rate-limited and scoped (e.g., "max 50 uses", "expires in 7 days").
- **Join Flow:**
  - Free trips: Auto-join if under limit.
  - Pro/Event trips: Join requests queue in `join_requests` table (with cooldowns to prevent spam) pending Admin approval.
- **Account Creation via Invite:** Tokenized invite state stored in `localStorage`, resolved securely after OAuth callback via `accept_invite` RPC.

---

## 8. Realtime + Sync Constitution

- **Chat & Messaging:** 100% delegated to Stream Chat. Supabase triggers for legacy chat broadcasts must be disabled to stop DB IO abuse.
- **Shared Objects (Tasks/Calendar):** Supabase Realtime using broadcast/postgres_changes.
- **Sync Throttling:** Client must debounce realtime events (e.g., wait 300ms before refetching collection) to prevent thundering herds when an Admin bulk-updates a Pro trip.

---

## 9. Scale-Tier Architecture Plan

- **Stage A (100-1K users):** Single DB, Stream for Chat, basic Supabase Realtime. *We are here.*
- **Stage B (1K-10K users):** Implement strict paginated lists, enforce OCC versioning across all objects, rate-limit Edge Functions.
- **Stage C (10K-100K users):** Read replicas for Event-Wide data (public itineraries). Redis/Upstash for invite link rate-limiting and session tracking.
- **Stage D (100K-1M+ users):** Sharded trip databases (active vs archived), dedicated realtime node scaling, fully separated microservices for multimodal AI processing.

---

## 10. Free vs Paid QoS Constitution

- **Free Tier:**
  - Best-effort TTS/AI.
  - Strict limits on trip size (max 20 users).
  - Synchronous media uploads (max 10MB).
- **Pro/Enterprise Tier:**
  - High-priority background queue for AI summary and multimodal parsing.
  - Bypass free-tier API rate limits.
  - Event trip capacity up to 5,000+ users.
  - Asynchronous background uploads with presigned URLs.

## 11. Dangerous Surface Ranking

1. **Legacy Channel Sync Triggers (Severity: Critical)**
   - *Failure Shape:* DB Lock contention, cascading trigger loops.
   - *Blast Radius:* Entire DB goes down on mass role change.
   - *Rule:* Migrate all sync logic to async background workers or Stream Webhooks.
2. **AI Mass-Import Functions (Severity: High)**
   - *Failure Shape:* OOM errors, Timeout, duplicate records on retry.
   - *Blast Radius:* User quota exhausted, corrupted trip state.
   - *Rule:* Idempotency keys + transaction boundaries required.
3. **Free-Tier Edge Function Rate Limiting (Severity: High)**
   - *Failure Shape:* Financial DoS (Billing exhaustion).
   - *Blast Radius:* GCP/OpenAI bill spikes.
   - *Rule:* Upstash Redis global rate-limiting by `authUserId`.
4. **Unversioned Basecamp Updates (Severity: Medium)**
   - *Failure Shape:* Two admins save simultaneously, silent overwrite.
   - *Blast Radius:* Lost itinerary text.
   - *Rule:* OCC Versioning required.

---

## 12. Recommended Immediate Platform Changes

1. **Drop Redundant Triggers:** Drop `sync_channel_on_role_change` to stop DB churn.
2. **Enforce Idempotency Middleware:** Add a shared Edge Function wrapper that checks `idempotency_keys` in Redis/DB before execution.
3. **Standardize OCC:** Apply the `version` integer check to `Basecamp`, `Trip Cover`, and `Explore Links`.
4. **Strict RLS on Artifacts:** Ensure `trip_artifacts` multimodal retrieval is locked down to specific `artifact_type` and `trip_id` to prevent cross-trip context bleed.

---

## 13. Exact Platform Changes

- **Code Areas:**
  - `src/utils/concurrencyUtils.ts`: Expand OCC hooks.
  - `supabase/functions/_shared/`: Introduce rate-limiting and idempotency middleware.
- **Schema/Index/Policy:**
  - `ALTER TABLE trips ADD COLUMN basecamp_version INT DEFAULT 1;`
  - Drop obsolete `trip_channel_members` table and related sync triggers.
  - Add `btree` index on `idempotency_key` across `trip_events`, `trip_tasks`, `trip_polls`.
- **Migration Order:**
  1. Add new columns/indexes. 2. Update RPCs/Edge Functions. 3. Update Client hooks. 4. Drop legacy triggers.
- **Rollback Plan:** Non-destructive DB migrations (always add columns, never drop without a 30-day deprecation window). Reverse proxy routing for Edge Function rollbacks.

## 14. Verification + Load Plan

- **Concurrency Testing:** K6 scripts firing 100 concurrent `update_task` requests for the same `task_id`; assert exactly 1 succeeds and 99 fail with 409.
- **Scale Testing:** Seed a trip with 5,000 members. Trigger a "Broadcast Announcement" and ensure DB CPU does not spike over 60% (validating Stream delegation).
- **Integrity Checks:** Daily cron script validating that `channel_members` counts match Stream channel counts.
